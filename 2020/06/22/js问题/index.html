js问题

发表于 2020-06-22 | 分类于 js

JavaScript 是单线程、异步、非阻塞、解释型脚本语言。
但实际上浏览器环境中还存在其他的线程，例如处理AJAX，DOM，定时器
同时浏览器中还维护了一个消息队列，主线程会将执行过程中遇到的异步请求发送给这个消息队列，等到主线程空闲时再来执行消息队列中的任务。

理解

浏览器组成：

1. 用户界面：地址栏，前进后退，书签等
2. 浏览器引擎：用户界面与渲染引擎之间传送指令
3. 渲染引擎：负责显示请求的内容。负责解析HTML和CSS内容，并展示内容在屏幕上
4. 网络：用于网络调用，比如HTTP请求，
5. 用户界面后台
6. JavaScript解释器：用于解释，执行js代码
7. 数据存储：

内核主要两部分：渲染引擎（Rendering Engine）和 js引擎
由于JS引擎越来越独立，浏览器内核 就倾向于 单指 渲染引擎。

渲染引擎

渲染流程：
从网络获取请求文档的内容
1. （Parser）解析HTML文档，将文档中的标签转为DOM节点树（内容树）
2. （）同时他会解析CSS文件以及style标签中的样式数据生成( CSS Rule Tree)，和DOM树一起构建出渲染树（render树）
渲染树是由一些带有视觉属性（color）的矩阵组成
3.（Layout） 布局，也叫回流，每一个节点分配一个屏幕坐标
4. （Paint）绘制，遍历render树，绘制每一个节点

渲染流程是逐步完成的，解析一部分，显示一部分内容，同时可能通过网络下载其余部分内容

渲染遇到JS文件

  1. js加载，解析和执行会阻塞DOM的构建，当遇到js文件DOM构建会暂停，等js引擎运行完毕后在恢复DOM构建，这也是为什么要把js放在body标签底部，也可以给script标签添加defer,async属性
  2. js改变css时，cssDOM也会阻塞DOM构建，不完整的CSSDOM是无法使用的，js访问CSSDOM并更改，那么执行js时，必须拿到完整的CSSDOM，所以当浏览器尚未完成CSSDOM，而我们却想运行脚本时，浏览器就会延迟脚本执行和DOM树构建，直到CSSDOM下载和构建完。所以浏览器会先下载和构建CSSOM，然后再执行JavaScript，最后在继续构建DOM。

重绘，回流

重绘：当render tree中的一些元素需要更新属性，而这些属性只是影响元素的外观、风格，而不会影响布局的，比如background-color。
回流:当render tree中的一部分(或全部)因为元素的规模尺寸、布局、隐藏等改变而需要重新构建
添加删除元素
元素尺寸变化
浏览器窗口变化，resize事件

async，defer

js操作DOM，其实涉及到两个线程之间的通信（渲染引擎，js引擎）,会带来新能上的损耗，并且操作DOM会导致重绘回流的情况

调用栈

代码在栈帧中执行。调用栈中顺序执行主线程代码，当调用栈中为空时，js引擎回去消息队列去取消息，取到后执行

浏览器提供的API

setTimeOut等
它们其实是在浏览器中提供的，也就是运行时提供的，因此，实际上除了JavaScript引擎以外，还有其他的组件。
其中有个组件就是由浏览器提供的，叫Web APIs，像DOM，AJAX，setTimeout等等。
还有就是非常受欢迎的事件循环和回调队列。
运行时负责给引擎线程发送消息，只负责生产消息，不负责取消息。

消息队列，事件循环（event loop）

JavaScript 是单线程、异步、非阻塞、解释型脚本语言。
但实际上浏览器环境中还存在其他的线程，例如处理AJAX，DOM，定时器
同时浏览器中还维护了一个消息队列，主线程会将执行过程中遇到的异步请求发送给这个消息队列，等到主线程空闲时再来执行消息队列中的任务。
同步任务的缺点是阻塞，异步任务的缺点是会使代码执行顺序难以判断

主线程在执行过程中遇到了异步任务，就会去注册这个函数，通过event loop线程通知相应的工作线程（如ajax，dom，setTimout等），同时主线程继续向后执行，不会等待。等到工作线程完成了任务，eventloop线程会将消息添加到消息队列中，如果此时主线程上调用栈为空就执行消息队列中排在最前面的消息，依次执行。
新的消息进入队列的时候，会自动排在队列的尾端。
单线程意味着js任务需要排队，如果前一个任务出现大量的耗时操作，后面的任务得不到执行，任务的积累会导致页面的“假死”。这也是js编程一直在强调需要回避的“坑”。
主线程会循环上述步骤，事件循环就是主线程重复从消息队列中取消息、执行的过程。
需要注意的是 GUI渲染线程与JS引擎是互斥的，当JS引擎执行时GUI线程会被挂起，GUI更新会被保存在一个队列中等到JS引擎空闲时立即被执行。因此页面渲染都是在js引擎主线程调用栈为空时进行的。

事件触发线程 同样是浏览器渲染引擎提供的，它会维护一个 消息队列。

JS引擎线程遇到异步（DOM事件监听、网络请求、setTimeout计时器等…），会交给相应的线程单独去维护异步任务，等待某个时机（计时器结束、网络请求成功、用户点击DOM），然后由 事件触发线程 将异步对应的 回调函数 加入到消息队列中，消息队列中的回调函数等待被执行。

同时，JS引擎线程会维护一个 执行栈，同步代码会依次加入执行栈然后执行，结束会退出执行栈。
如果执行栈里的任务执行完成，即执行栈为空的时候（即JS引擎线程空闲），事件触发线程才会从消息队列取出一个任务（即异步的回调函数）放入执行栈中执行。

cookie , session , token

以前web基本只是文档的浏览，不需要记录谁在哪段时间浏览了哪些文档，每个请求都是全新的
交互式的web应用，要管理会话，必须记住哪些人登录系统，把每个人区分开，HTTP请求是无状态的，

cookie

cookie是浏览器里面能永久存储的一种数据（他是客户端无状态协议的一种解决方案）

cookie原理

  1. 客户端第一次请求，发送数据到服务器
  2. 服务器返回响应信息的同时，还会传回一个cookie，
  3. 客户端接收到服务器响应之后，浏览器会将cookie保存
  4. 客户端在此发起请求时，会携带cookie

由于cookie在存在客户端，所以浏览器加入了一些限制确保cookie被恶意修改

与Storage区别

HTML5提供了
SessionStorage：仅当前会话下有效，关闭页面清楚，仅在客户端保存，不参与和服务器通讯
LocalStorage：手动清楚，否则一直存在，仅在客户端保存，不参与和服务器通讯
cookie：4k左右，一般由服务器生成，每次都会携带http头中（同源策略），

生命周期

整个会话期间：浏览器关闭时自动删除

session（保存在服务器）

session原理

  1. 服务器在处理客户端请求的过程中会创建session，并未该session生成唯一session id（这个id就是在请求中会被用来重新获取已经创建的session）
  2. 服务器将session id发送给客户端
  3. 当客户端再次请求时，会带上session id
  4. 服务器接收到请求，更具id去查找session

    生命周期

    与cookie一样，服务器也可以设置session的生效时间

服务器要知道当前请求是谁发送给我的，为了做这种区分，服务器就是要给每个客户端分配不同的”身份标识”，然后客户端每次向服务器发请求的时候，都带上这个”身份标识“，服务器就知道这个请求来自与谁了

token

token原理

  1. 客户端第一次请求，发送用户信息到服务器，服务器将签名和数据一起作为token返回给客户端
  2. 服务器不保存token，客户端保存token
  3. 当用户再次发起请求时，在请求信息中将token一起发送给服务器
  4. 服务器用同样的HSA256算法和密钥，对数据再计算一次签名，和token的签名做比较
  5. 如果相同，服务器就知道客户端登录过，则反之

区别

cookie，session

  1. 存储位置不同
  2. cookie不是很安全，被人可以分析本地的cookie进行cookie欺骗
  3. session会在一定时间保存在服务器上，访问增多，就会比较占用服务器性能
  4. 存储大小不同：cookie不超过4k，并且保存的是字符串，很多浏览器限制一个站点最多保存20个，session保存的是对象
    一般建议： 将登陆信息等重要信息存放为session, 其他信息如果需要保留，可以放在cookie中

Token，session

  1. session是一种HTTP存储机制，为无状态HTTP提供持久机制
  2. Token就是令牌，比如你授权登录时，他就是依据，判断你是否已经授权该软件
    Session和Token并不矛盾，作为身份认证Token安全性比Session好，因为每一个请求都有签名还能防止监听以及重放攻击，
    而Session就必须依赖链路层来保障通讯安全了。如上所说，如果你需要实现有状态的回话，仍然可以增加Session来在服务端保存一些状态。

线程，进程

一个程序至少有一个进程，一个进程至少有一个线程
线程的划分尺度小于进程，使得多线程程序并发性高
进程在执行过程中拥有独立的内存单元，而多线程共享内存，从而极大的起高了程序的运行效率

区别

  1. 每一个独立的线程有一个程序的入口，顺序执行序列和进程的出口，但是线程不能够独立执行，必须依存在应用程序中，有应用程序提供多个线程执行控制
  2. 从逻辑角度看，多线程的意义在于一个应用程序中，有多个执行部分可以同时执行
  3. 操作系统并没有将多个线程看做多个独立的应用，来实现进程的调度和管理以及资源分配

闭包概念

f2就是闭包，闭包就是能够读取其他函数内部变量的函数，定义在一个函数内部的函数，闭包就是将函数内部和函数外部连接起来的一座桥梁

闭包用途

  1. 可以读取函数内部的变量，
  2. 让这些变量的值始终保持在内存中
    回调，setTimeout()，函数防抖，封装私有变量

    注意

  3. 由于闭包会使得函数中的变量都被保存在内存中，内存消耗很大，所以不能滥用闭包，否则会造成网页的性能问题，在 IE 中可能导致内存泄露。解决方法是，在退出函数之前，将不使用的局部变量全部删除。
  4. 闭包会在父函数外部，改变父函数内部变量的值。所以，如果你把父函数当作对象（object）使用，把闭包当作它的公用方法（Public Method），把内部变量当作它的私有属性（private value），这时一定要小心，不要随便改变父函数内部变量的值。

ES5,ES6两者继承

ES5继承通过prototype，或构造函数机制
S6 通过 class 关键字定义类，里面有构造方法，类之间通过 extends 关键字实现继承。子类必须在 constructor 方法中调用 super 方法，否则新建实例报错。因为子类没有自己的 this 对象，而是继承了父类的 this 对象，然后对其进行加工。如果不调用 super 方法，子类得不到 this 对象。

ES5 的继承实质上是先创建子类的实例对象，然后再将父类的方法添加到 this 上，Parent.apply(this)
ES6 的继承机制完全不同，实质上是先创建父类的实例对象 this（所以必须先调用父类的 super()方法），然后再用子类的构造函数修改 this
ps：super 关键字指代父类的实例，即父类的 this 对象。在子类构造函数中，调用 super 后，才可使用 this 关键字，否则报错。

节流（throttle），防抖（debounce）

函数防抖

在事件被触发n秒后再执行回调，如果在这n秒内又被触发，则重新计时。
加入了防抖以后，当你在频繁的输入时，并不会发送请求，只有当你在指定间隔内没有输入时，才会执行函数。如果停止输入但是在指定间隔内又输入，会重新触发计时

函数节流

规定在一个单位时间内，只能触发一次函数。如果这个单位时间内触发多次函数，只有一次生效。

总结

函数防抖和函数节流都是防止某一时间频繁触发，但是这两兄弟之间的原理却不一样。
函数防抖是某一段时间内只执行一次，而函数节流是间隔时间执行。
